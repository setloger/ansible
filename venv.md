## Структура хорошего проекта с venv

```text
ansible-project/
├── requirements.txt        # pip-зависимости (ansible-core, jinja2...)
├── requirements.yml        # ansible-galaxy коллекции
├── .venv/                  # сам venv (в .gitignore!)
├── ansible.cfg
├── inventory/
└── playbooks/
```

### Онбординг нового участника команды

Три команды:

- python3 -m venv .venv && source .venv/bin/activate
- pip install -r requirements.txt
- ansible-galaxy collection install -r requirements.yml

---

## Требования

Перед началом убедитесь, что у вас есть:

- Ubuntu/Debian
- Python 3.10+
- Ansible-проект в git-репозитории

---

## 1. Установка python3-venv

На Debian/Ubuntu модуль `venv` часто ставится отдельным пакетом:

```bash
sudo apt install python3.10-venv
```

---

## 2. Создание виртуального окружения

Перейдите в каталог проекта и создайте локальное окружение:

```bash
cd ~/ansible
python3 -m venv .venv
```

После этого в проекте появится папка `.venv/` с изолированным Python и собственным `pip`.

---

## 3. Активация окружения

Активируйте созданный `venv`:

```bash
source .venv/bin/activate
```

Если всё прошло успешно, в начале строки терминала появится префикс:

```bash
(.venv)
```

Это значит, что команды `python`, `pip` и `ansible` будут использоваться именно из локального окружения.

---

## 4. Создание requirements.txt

Создайте файл зависимостей:

```bash
nano requirements.txt
```

Добавьте в него следующий набор пакетов:

```txt
ansible-core==2.17.10
Jinja2==3.1.6
MarkupSafe==2.1.5
cryptography==46.0.6
PyYAML==6.0.3
resolvelib==1.0.1
packaging==24.2
python-dateutil==2.9.0.post0
six==1.17.0
```

---

## 5. Установка зависимостей

Установите зависимости из файла:

```bash
pip install -r requirements.txt
```

---

## 6. Проверка установки

Проверьте, что `ansible` используется именно из `.venv`, а не из системного окружения:

```bash
which ansible
```

Ожидаемый путь будет примерно таким:

```bash
/home/<user>/aansi/.venv/bin/ansible
```

Проверьте версию Ansible:

```bash
ansible --version
```

Ожидаемо должна отображаться версия:

```bash
ansible [core 2.17.10]
```

Дополнительно можно посмотреть список всех установленных пакетов:

```bash
pip list
```

---

## 7. Добавление .venv в .gitignore

Папку виртуального окружения не нужно коммитить в репозиторий.

Добавьте её в `.gitignore`:

```bash
echo '.venv/' >> .gitignore
```

---

## 8. Автоактивация через direnv (опционально)

Если хочется, чтобы окружение активировалось автоматически при входе в папку проекта, можно использовать `direnv`.

### Установка direnv

```bash
sudo apt install direnv
```

### Подключение к zsh

```bash
echo 'eval "$(direnv hook zsh)"' >> ~/.zshrc
source ~/.zshrc
```

### Создание `.envrc`

```bash
echo 'source .venv/bin/activate' > .envrc
direnv allow .
```

Теперь при переходе в каталог проекта `venv` будет активироваться автоматически.

---

## Повседневная работа

Обычный рабочий сценарий:

```bash
cd ~/ansible
source .venv/bin/activate
ansible-playbook playbook.yml
deactivate
```

---

## Если нужно пересоздать окружение с нуля

Если окружение сломалось или нужно пересобрать его заново:

```bash
deactivate
rm -rf .venv
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

---

## Итог

После этих шагов у проекта будет локальное, изолированное Python-окружение для работы с Ansible. Это удобно для повторяемой установки зависимостей, чистой разработки и исключения конфликтов с системными пакетами.

Коротко

| Без venv                            | С venv                        |
| ----------------------------------- | ----------------------------- |
| "Работает только у меня"            | Одинаково у всех              |
| Обновление ломает другие проекты    | Проекты изолированы           |
| Непонятно какие версии используются | pip freeze даёт точный список |
| Сложно откатиться                   | Удалил папку — чисто          |
| CI/CD нестабильно                   | Воспроизводимые сборки        |
